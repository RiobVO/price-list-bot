# 0006. Стратегия callback-id

## Статус
Accepted

## Контекст
callback_data Telegram ограничен 64 байтами. У категории/подкатегории в данных нет id — только имя (произвольная UTF-8 строка, часто узбекская). Голый индекс позиции протухает при правке таблицы; сырое имя вылетает за лимит. Подкатегория «Прочее»/«0.5л» повторяется в разных категориях.

## Решение
Все callback-токены — 12-hex blake2s (digest_size=6) от нормализованного канона имени: категория, подкатегория И товар (id товара клиента может быть длинным). Подкатегория хешируется от пары (category, subcategory) с разделителем \x1f — глобально уникальна, одноимённые подкатегории не склеиваются. bot кладёт полученный id дословно и возвращает его в services; хеширование и обратные карты — только в services. Протухший id (любого уровня) → ViewResult.Stale (мягкое «каталог обновился» + меню).

## Последствия
Фикс. длина id (12 байт) → callback p:<prod>:<sub>:<page> ≈ 32 байта, с запасом ≤ 64. id стабилен между снимками к смене регистра/письменности. Коллизия хешей (разные имена → один id) — known limitation: error-лог, первый wins (48-битный blake2s на ~50 группах делает это практически невозможным).

## Альтернативы
ASCII-slug транслит — нестабилен на UZ кириллица/латиница, длинный, коллизии. Индекс позиции — массовое тихое протухание при правке таблицы. Оба отвергнуты.
