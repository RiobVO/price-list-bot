# 0012. Граница services: view-модели и ViewResult

## Статус
Accepted

## Контекст
Слою bot (aiogram) нужны данные каталога для клавиатур и сообщений. Инвариант: services не импортирует aiogram. Нужно решить, что именно services отдаёт хендлерам и как сигнализирует протухший callback-id.

## Решение
services отдаёт презентационно-готовые view-модели (frozen dataclass: CategoryItem/SubcategoryItem/ProductListItem/Page/ProductCard) с уже выбранным языком, фолбэком desc, форматированной ценой и обрезанным caption. bot — тонкий транспорт «view → клавиатура», хеш id кладёт в callback дословно. Исход запроса по id — единый union ViewResult = Ok | Stale (протухший id → Stale → мягкое «каталог обновился» + меню). Ядро (навигация/поиск/пагинация/форматирование) — чистые функции над иммутабельным Catalog; поверх — тонкий cache-aware фасад CatalogService, читающий снимок один раз на вызов и мемоизирующий CatalogIndex по идентичности снимка.

## Последствия
Вся презентация без aiogram тестируется на статических Catalog-фикстурах. bot тонкий. Консистентность снимка в пределах клика гарантирована (один read). Протухание — нормальный поток, а не исключение.

## Альтернативы
Отдавать сырые Product и форматировать в bot — размазывает презентацию по aiogram-слою, тяжелее тестировать, расходится со скоупом services. Отвергнуто. Исключение для протухания вместо Ok/Stale — control-flow через exception, легко забыть except. Отвергнуто.
