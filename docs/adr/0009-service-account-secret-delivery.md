# 0009. Доставка секрета сервис-аккаунта Google

Статус: Accepted

## Контекст

Боту нужен JSON-ключ сервис-аккаунта Google, чтобы читать таблицу через gspread. Это файл, а не
короткий токен. Способов доставки в контейнер несколько (смонтированный файл, переменная окружения,
секрет-менеджер), и они по-разному ложатся на деплой клиента. Сам JSON не должен попадать в репозиторий
(бриф §7). Нужно зафиксировать один интерфейс, не плодя ветвления по коду.

## Решение

- **РОВНО один способ за запуск**, выбор — в конфиге:
  - `GOOGLE_APPLICATION_CREDENTIALS` — путь к смонтированному JSON-файлу (дефолт; Docker volume/secret);
  - `GOOGLE_CREDENTIALS_B64` — base64 от JSON-содержимого (альтернатива для платформ «секрет = env-строка»).
- Валидатор `Settings` требует, чтобы был задан ровно один из двух (оба пусты ИЛИ оба заданы → ошибка).
- Сборка клиента изолирована в `src/data/auth.py` (`build_gspread_client`): путь → `gspread.service_account`,
  base64 → декод в dict → `gspread.service_account_from_dict`. Битый base64 → `ValueError` на старте.
- Секреты типизированы `SecretStr`; `.env` и `*service-account*.json` — в `.gitignore`.

## Последствия

- Один предсказуемый интерфейс; клиент выбирает способ под свою платформу, код не ветвится в рантайме.
- gspread импортируется только в `auth.py`/`fetch.py` — граница грязного I/O сохранена.
- base64 устраняет проблемы с переносами строк/экранированием при доставке JSON через env.
- Ошибка доставки секрета проявляется на старте (`ValueError`/non-transient → `exit(1)`), а не молча.

## Альтернативы (отвергнуты)

- **Только env-JSON (raw)**: переносы строк и кавычки JSON ломаются в большинстве env-механизмов.
- **Только путь к файлу**: не подходит платформам, где секрет доступен лишь как env-строка.
- **Поддержка обоих одновременно с приоритетом**: неявная «какой победит» логика; явное «ровно один»
  проще и диагностируемее.
