## 0. Как читать этот бриф (контракт исполнителя)

Этот документ — контракт, а не вдохновение. Правила интерпретации:

- Что помечено **«зафиксировано»** или **«жёстко»** — инвариант. Нарушение = баг, а не «вариация».
- Пробел в брифе **не закрывается догадкой**. Нашёл дыру — выноси в §9 «Открытые вопросы», не выдумывай факт.
- Где по необходимости принял допущение — помечай его дословно `ДОПУЩЕНИЕ:` в коде (комментарий) и в README. Допущение ≠ решение клиента.
- Порядок работы: слой `data` строится **через тесты** (TDD). Сначала тест парсера на статической фикстуре строк, потом парсер. Слой `data` — каркас всего, ошибка здесь течёт во всё остальное.
- Если правка цены или товара требует трогать код — решение неверное (см. §1, главный принцип).

## 1. Контекст и цель

Компания-производитель/дистрибьютор хочет Telegram-бота, который работает как
**интерактивный двуязычный прайс-лист**. Клиент сам ведёт ассортимент и цены в
Google Sheets; бот автоматически отражает изменения. Оформления заказов **нет** —
бот только показывает товары.

**Главный принцип проекта:** заказчик должен иметь возможность менять цены и товары
без участия разработчика. Любое решение, которое требует правки кода при изменении
цены — неверное.

## 2. Скоуп (зафиксировано с клиентом)

**В скоупе:**
- Двуязычный интерфейс: узбекский (`uz`) и русский (`ru`). Переключение языка пользователем.
- Источник данных — **Google Sheets** (клиент редактирует сам).
- Карточка товара содержит: название, описание, цена опт, цена розница, фасовка, фото.
- ~500 наименований → **обязательны**: категории, поиск по названию, пагинация.
- Кэширование данных из Sheets (нельзя дёргать API на каждый запрос).

**Вне скоупа (НЕ реализовывать):**
- Оформление заказов, корзина, оплата.
- Админ-панель (управление — через саму Google-таблицу).
- Аналитика, рассылки, CRM-интеграции.

Если по ходу возникнет соблазн добавить что-то из «вне скоупа» — остановись и
вынеси это в раздел «Открытые вопросы», не реализуй молча.

## 3. Технологический стек

- **Язык/фреймворк:** Python 3.11+, `aiogram` 3.x (async).
- **Данные:** Google Sheets через `gspread` + service-account. Id книги (spreadsheet) и имя листа —
  **в конфиге (env), не хардкод**: клиент вправе переименовать вкладку, это не должно ронять бота (§1).
- **Кэш:** in-memory с TTL (обновление раз в N минут, N в конфиге). Без БД на старте.
- **Конфиг:** все секреты и параметры через переменные окружения (`.env`, `pydantic-settings`).
- **Секрет service-account:** это JSON-файл, а не строка-токен. Зафиксируй один способ доставки:
  по умолчанию — **путь к файлу**, смонтированному в контейнер (`GOOGLE_APPLICATION_CREDENTIALS`
  через Docker volume/secret); JSON-в-одной-переменной (base64) — опционально, под тем же интерфейсом.
  Сам JSON в репозиторий не попадает никогда (см. §7). Способ подтвердить в §9.
- **Деплой:** Docker-контейнер, long-polling на старте (webhook — опционально, под флагом).
  Контейнер обязан корректно реагировать на `SIGTERM` (graceful shutdown — см. §7).
- **Качество:** `ruff` (lint+format), `mypy` (strict), `pytest`.

Если предлагаешь отклонение от стека — обоснуй в «Открытых вопросах» до реализации.

## 4. Модель данных (Google Sheets → бот)

Таблица — это контракт между клиентом и ботом. Структура листа `products`:

| Колонка | Ключ | Тип | Обязательна | Примечание |
|---|---|---|---|---|
| ID | `id` | str/int | да | Уникальный, неизменный идентификатор товара |
| Категория | `category` | str | да | Группировка в меню |
| Название (RU) | `name_ru` | str | да | |
| Название (UZ) | `name_uz` | str | да | |
| Описание (RU) | `desc_ru` | str | нет | |
| Описание (UZ) | `desc_uz` | str | нет | |
| Цена опт | `price_wholesale` | number | да | Валюта/формат — см. ниже и §9 |
| Цена розница | `price_retail` | number | да | |
| Фасовка | `packaging` | str | нет | |
| Фото | `photo` | str (URL или file_id) | нет | См. «Открытые вопросы» |
| Активен | `is_active` | bool | да | Скрытие товара без удаления строки |

**Сопоставление колонок — по именам заголовков, не по позиции.** Парсер находит колонки по
ключам из строки заголовка (первая строка листа), а не по индексу. Клиент вставит/переставит
колонку или добавит свою — каталог не должен сломаться. Лишние неизвестные колонки игнорируются.

**Валюта и формат цены.** В модели цена — голое число; валюта и формат вывода в модели не заданы.
`ДОПУЩЕНИЕ` на старте: одна валюта на весь каталог, задаётся в конфиге (env); форматирование
(разделитель тысяч, символ/код валюты, позиция) — через i18n-словарь, единообразно для всех цен.
Это допущение, не факт — подтвердить в §9 (мультивалютность сюда не закладываем, пока не спрошено).

**Правила парсинга (жёсткие):**
- Парсер слоя `data` — **чистая функция** вида `Sequence[Mapping[str, str]] -> Catalog`: на вход
  сырые строки (уже прочитанные из Sheets), на выходе валидированный каталог. Парсер **не знает**
  про `gspread`, сеть и aiogram. Это разделяет «получить строки» (грязный I/O) и «разобрать строки»
  (чистая логика) — и делает §8-тесты возможными без обращения к Google API.
- **Битая строка ≠ битая схема. Различай явно:**
  - *Битая строка* (пустое обязательное поле, кривое число у отдельного товара) → строку пропустить
    или деградировать (см. ниже), залогировать `warning`, **остальной каталог жив**.
  - *Битая схема* (нет обязательной колонки в заголовке, лист пуст/переименован) → это сломанный
    контракт, а не плохая строка. Снимок **отвергается целиком**, логируется `error`, старый кэш
    остаётся в силе (см. §7, атомарность). Одна опечатка в заголовке не должна обнулить магазин.
- Строки с пустыми обязательными полями → пропустить и залогировать (НЕ падать).
- Невалидная цена (не число) → товар помечается как «цена по запросу», не падаем.
- **Толерантный разбор типов** (клиент — живой человек, ведущий таблицу руками):
  - `number`: принимать пробелы-разделители тысяч и запятую как десятичную (`120 000`, `12,5`).
  - `bool` (`is_active`): принимать чекбокс Sheets (`TRUE/FALSE`) и человеческие варианты
    (`1/0`, `да/нет`, `ha/yo'q`, `+/-`, регистронезависимо). Нераспознанное непустое значение → `warning`.
    Пустое `is_active` — обязательное поле: по общему правилу строка пропускается (но см. §9 — у этого
    есть UX-риск: забытая галочка спрячет товар). Молчаливый дефолт не вводим.
- Дубликаты `id` → берётся первый, остальные логируются как `warning`.
- Бот ВСЕГДА переживает кривую таблицу: плохие данные деградируют gracefully, не роняют процесс.

## 5. Поведение бота (UX)

- `/start` → выбор языка (если не выбран ранее) → главное меню.
- Главное меню: список категорий (inline-кнопки) + кнопка «Поиск» + кнопка «Сменить язык».
- Категория → пагинированный список товаров (inline, N на страницу, кнопки «◀ / ▶»).
- Товар → карточка: фото + название + описание + опт/розница + фасовка. Кнопка «Назад».
- Поиск: пользователь вводит текст → бот ищет по `name_{lang}` (регистронезависимо, по подстроке). Перед сравнением — нормализация строк (lower + trim + приведение узбекских диакритик `oʻ→o`, `gʻ→g` и унификация кириллица/латиница, насколько применимо). Без нормализации поиск на узбекском будет промахиваться.
  - **Нормализация узбекского — не один-два символа, а таблица соответствий.** В UZ две письменности
    параллельно (латиница ↔ кириллица: `oʻ/gʻ/q/h ↔ ў/ғ/қ/ҳ` и т.д.) плюс варианты апострофа (`ʻ` U+02BB
    против обычного `'`). Реализуй как явную таблицу + функцию нормализации с юнит-тестами; не считай
    тривиальным. Цель: запрос на любой письменности находит товар, чьё `name_uz` ведётся на другой.
- **Языковой фолбэк:** если поле на выбранном языке пустое (напр. `name_uz` пуст у пользователя на `uz`) — показывать значение на втором языке, а не пустоту.
- **Лимит `callback_data` = 64 байта (Telegram Bot API) — это проектное ограничение клавиатур:**
  в `callback_data` класть только короткие идентификаторы (числовой/короткий id категории, номер
  страницы), **не** сами строки (узбекские названия категорий в UTF-8 легко вылетят за лимит) и
  **не** поисковый запрос. Поиск — через FSM-состояние (ввод текста), а не через callback.
- **Любой callback-id может протухнуть** (категория, товар, страница). Маппинг `id ↔ объект` живёт в
  снимке кэша, а источник — строки, которые клиент вправе переименовать/удалить между снимками.
  Стабильный id (slug/хеш ключа, не голый индекс) снижает частоту протухания, но **не устраняет** его.
  Инвариант, единый для категорий, карточек товара и страниц пагинации: callback с неизвестным/
  протухшим id → мягкий ответ «каталог обновился, откройте заново» + возврат в меню, не падать и не
  отвечать пустотой. Карточка товара по исчезнувшему `id` — тот же случай, не исключение.
- **FSM-состояние сбрасывается при выходе из поиска:** нажал категорию/«Назад», находясь в режиме
  ввода поиска — состояние снимается, бот не «застревает» в ожидании текста.
- **Фото может не отправиться** (URL отдал 404 / не картинку / превысил лимиты Telegram на фото).
  Это не ошибка хендлера: отдать карточку **без фото** + текст, залогировать `warning`. Карточка
  всегда доходит, даже если картинка битая (тот же принцип «переживаем кривые данные», но на доставке).
- **Лимит подписи к фото = 1024 символа (Telegram Bot API)** — отдельный от лимита сообщения (4096).
  Карточка (фото + caption) не должна превышать его: длинное `desc` обрезать с многоточием по границе,
  либо слать фото и текст двумя сообщениями. Дефолт — обрезка `desc` в caption; не падать на длинном описании.
- **Пустые состояния определены явно, не пустой экран:** поиск без результатов → «ничего не найдено»
  + кнопка нового поиска/назад; категория без активных товаров → «в этой категории пока нет товаров»;
  каталог пуст целиком (cold-start/деградация) → «каталог обновляется» (§7).
- Язык хранится per-user (in-memory на старте; задокументируй, что при рестарте сбросится — см. «Открытые вопросы»).
- Все пользовательские тексты — через i18n-словарь (`locales/ru.py`, `locales/uz.py`), без хардкода строк в логике.

## 6. Архитектура (Clean-ish, без оверинжиниринга)

```
src/
  bot/            # хендлеры aiogram, клавиатуры, FSM-состояния
  services/       # бизнес-логика: каталог, поиск, пагинация
  data/           # gspread-клиент (fetch) + парсинг (pure) + кэш с TTL
  locales/        # ru.py, uz.py — i18n словари
  config.py       # pydantic-settings, чтение из env
  main.py         # точка входа, сборка зависимостей, запуск refresh-задачи
tests/
```

- Слой `data` не знает про aiogram. Слой `bot` не лезет в gspread напрямую — только через `services`.
- **Внутри `data` разделены две ответственности:** `fetch` (грязный I/O — `gspread`, сеть, квоты)
  и `parse` (чистая функция строки→каталог, §4). Тестируется `parse`, а не сеть.
- **Нормализация типов — на границе `fetch`.** `gspread` отдаёт `Any` (числа/булевы родными типами);
  `fetch` приводит все ячейки к `str`, а `parse` принимает `Mapping[str, str]`. Так фикстуры в тестах
  (чистые строки) не расходятся с продакшеном, а `Any` не течёт в ядро логики под `mypy --strict`.
- Кэш — единый источник правды для хендлеров; обращений к Google API из хендлеров быть не должно.
- **Обновление кэша — фоновая `asyncio`-задача по TTL, а не ленивая подгрузка в хендлере.**
  Запись в кэш под `asyncio.Lock` (single-flight): один параллельный апдейт, без гонок и без
  thundering herd, когда TTL истёк и сразу пришло много запросов. Хендлеры всегда читают готовый
  снимок без ожидания сети. Задача стартует в `main.py`, отменяется при shutdown (§7).

## 7. Нефункциональные требования

- **Холодный старт (кэша ещё нет).** Если на старте Google API недоступен — кэша нет, отдавать
  нечего. Поведение: ретраи с backoff на первичную загрузку; пока валидного снимка нет — бот
  поднимается и отвечает «каталог обновляется», **не падает и не показывает пустоту как факт**.
  Сколько ретраить и что именно показывать — зафиксировать (см. §9).
- **Отказоустойчивость (рантайм).** Если Google API недоступен при обновлении — бот отдаёт последний
  валидный кэш и логирует ошибку, не падает.
- **Атомарность кэша.** Новый снимок заменяет текущий кэш только если он (а) прошёл проверку схемы и
  (б) распарсился целиком. Если чтение поймало таблицу в момент правки (полу-пустые строки массово)
  или схема сломана — старый кэш остаётся в силе, пишется `error`. Один кривой апдейт не должен
  обнулять каталог. Частично распарсенный снимок в кэш **не кладётся**.
- **Производительность:** карточка/список отвечают из кэша (без сетевого вызова к Sheets в горячем пути).
- **Троттлинг:** middleware-ограничение частоты на пользователя (защита от спама пагинацией/поиском). Дефолт — разумный лимит запросов в секунду на юзера.
- **Лимиты:** учесть квоты Google Sheets API (читать всю таблицу разом по TTL, не построчно).
- **Graceful shutdown.** На `SIGTERM`/`SIGINT`: отменить фоновую refresh-задачу, закрыть сессию бота,
  дождаться завершения текущих хендлеров. Никаких подвисших задач при рестарте/деплое.
- **Логирование:** структурные логи (старт, обновление кэша, ошибки парсинга, ошибки API). Полезные
  поля: время последнего успешного обновления кэша, число валидных/пропущенных строк в снимке.
- **Секреты:** ничего в репозиторий. `.env.example` — да, `.env` и JSON service-account — в `.gitignore`.

### Открытые вопросы (ответить/зафиксировать допущения до реализации)

- **Фото:** рекомендуемый вариант — **публичные URL** в таблице. Причина: Telegram `file_id` привязан к конкретному боту и клиент не сможет проставлять его сам, ведя таблицу. Реализация: если значение похоже на URL — грузим по URL; если это похоже на `file_id` — пробуем как `file_id` (фолбэк для ранее загруженных фото). Дефолт для клиента в README — только URL.
- **Валюта и формат цены:** одна валюта на каталог (из конфига) или возможна мультивалютность? Какой
  символ/код (сум/UZS), разделитель тысяч, позиция символа? До ответа — `ДОПУЩЕНИЕ` из §4.
- **Секрет service-account:** путь к смонтированному файлу (дефолт) или JSON-в-env? Подтвердить под деплой клиента.
- **Холодный старт:** сколько ретраить недоступный API на старте (предел/backoff) и какой текст
  показывать пользователю, пока каталог не загружен?
- **Память языка/состояния:** in-memory сбрасывается при рестарте. Достаточно ли для старта, или нужен лёгкий persist (SQLite/Redis)? По умолчанию: in-memory + TODO на persist.
- **TTL кэша:** дефолт 5 минут — ок? Клиент поменял цену, через сколько максимум это видно в боте?
- **Форс-рефреш:** нужна ли команда ручного обновления кэша (whitelist по `user_id`)? Это пограничный
  «мини-админ» — не реализую молча, выношу как вопрос. По умолчанию — нет, ждём TTL.
- **Категории:** плоский список или есть подкатегории? Модель сейчас рассчитана на плоский. Подтвердить.
- **Дефолт `is_active`:** сейчас пустое `is_active` = пропуск строки (общее правило обязательных полей).
  Не безопаснее ли дефолтить «виден», чтобы забытая галочка не прятала товар? Решение за клиентом.
- **Сколько товаров на странице** при пагинации (дефолт 8–10)?
- **Webhook vs polling** в проде — есть ли уже сервер/домен, или polling достаточно?

Не выдумывай ответы как факты. Где принял допущение — пометь его явно как допущение в коде/README.

## 10. Чего НЕ делать

- Не хардкодить товары, цены, строки интерфейса.
- Не парсить таблицу по индексам колонок — только по именам заголовков (§4).
- Не хардкодить id книги и имя листа — только из конфига (§3).
- Не ходить в Google API из хендлеров.
- Не обновлять кэш лениво в хендлере — только фоновая refresh-задача (§6).
- Не класть в кэш частично распарсенный или схемно-битый снимок (§7).
- Не класть строки/поисковый запрос в `callback_data` — только короткие id (§5).
- Не падать на кривых данных.
- Не тащить БД/Redis/очереди, пока они реально не нужны (см. скоуп).
- Не расширять скоуп молча.

## 11. Architecture Decision Records (ADR)

Значимые архитектурные решения фиксируй **по ходу разработки** — в момент выбора, отдельными
файлами `docs/adr/NNNN-title.md` (формат Nygard: Статус / Контекст / Решение / Последствия /
Альтернативы). ADR заводится на выбор с альтернативами и долгими последствиями (источник данных,
стратегия кэша, транспорт, границы слоёв), а не на каждый инвариант реализации. Задним числом
ADR не редактируют: отменённое решение помечается `Superseded` и заводится новый.
